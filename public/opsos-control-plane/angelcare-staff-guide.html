1. What the OPSOS Control Plane is

OPSOS is not a normal dashboard. It is AngelCare’s internal runtime governance layer: a place to observe the app, detect slow or risky behavior, activate safe controls, and record every operational decision.

Read first
◎

Observe

Understand pages, routes, modals, APIs, telemetry, errors, safe modes and live runtime conditions before acting.

⚙

Control

Use feature flags, runtime configs and safe modes to reduce risk without editing code or redeploying immediately.

↩

Audit & rollback

Every meaningful operation should be logged, attributable and reversible where technically possible.

2. Non-negotiable staff rules

The control plane is powerful. Never treat it like a playground. Every switch can affect a real user workflow.

Safety

Never do this

Do not enable global safe mode without approval. Do not run risky repair actions just to “see what happens.” Do not ignore warning modals. Do not operate outside your role.

Always do this

Read the diagnosis, run dry checks when available, confirm the scope, check rollback status, and write a clear reason before executing.

3. Roles and allowed behavior

OPSOS actions must be separated by responsibility. Staff can read and report; technical admins can operate; CEO/Super Admin can approve high-risk actions.

7. Modals & UX Failure Diagnosis

Many browser freezes happen when a modal mounts too much UI, keeps memory after close, or triggers repeated API calls.

Open-time diagnosis

Measure how long it takes from click to usable modal. Anything above 800ms should be inspected.

Retained state

If memory rises after closing a modal and never comes down, there may be retained state or listeners.

Print preview risk

Print/PDF previews should load on demand, not when the modal opens.

8. API & Queue Control

APIs should be checked for 401/403/500 errors, long durations, repeated calls and failed queue work.

SignalMeaningStaff action
500 errorsServer or database problemOpen API recovery, do not retry endlessly
Repeated fetchesPossible render/fetch loopDisable polling if scoped control exists
403 errorsRBAC or auth blockCheck user role before escalating
Slow endpointQuery/index/data volume issueRecord target and request technical review

9. Performance & memory console

Staff should understand long tasks, memory pressure and frozen interactions without guessing.

Long task

A browser task above 50ms can freeze interaction. Repeated long tasks usually mean too much JS/rendering.

Memory pressure

If memory increases when opening/closing the same modal repeatedly, the page may leak state or listeners.

10. Feature Flags

Feature flags are controlled runtime switches. They should change behavior safely without redeploying.

Important distinction

A flag is real only if the target page actually reads it. If a flag is staged, it is preparation only. Always check action result and target behavior.

11. Safe Mode Profiles

Safe mode is the emergency brake. It should reduce load, simplify UI and keep users operational.

Route safe mode

Limits a specific route such as Market-OS Campaign Lifecycle.

Modal safe mode

Limits a specific modal such as Employee360DossierModal.

Global safe mode

Emergency only. Requires approval and audit reason.

12. Repair Skills Library

Repair skills should never pretend. Each action must clearly be dry-run, staged, blocked, or executed.

Route Recovery

Inspect route risk, clear safe cache, revalidate where supported.

Memory Repair

Apply row limits, lazy modals, reduced polling and animation controls.

API Recovery

Test endpoints and queue states before retrying operations.

Rollback

Restore previous runtime configuration where rollback points exist.

14. Rollback Center

Rollback is not a panic button. It is a controlled return to a previous known state.

Rollback rule

Only rollback actions that have a stored rollback point. If rollback is unavailable, escalate instead of improvising.

15. Real workflows

These workflows teach staff exactly what to do in common situations.

1

Slow campaign lifecycle page

Open OPSOS → Pages & Routes → inspect /market-os/campaign-lifecycle → check cards/polling/memory → enable route safe mode if approved → verify page → log result.

2

Employee360 modal freezes

Open Modals & UX → select Employee360DossierModal → check print preview and retained memory → enable lazy print preview → limit history rows → re-test close/open.

3

API failures appear

Open API Control → inspect endpoint → check status and role errors → run dry health test → escalate 500s, fix RBAC for 403s, do not spam retry.

4

Need emergency safe mode

Confirm impact → select scoped safe mode → write reason → execute → verify target page → notify staff → plan rollback after stability returns.

16. Emergency protocol

When browser crashes or staff cannot operate, follow this protocol exactly.

Critical

Emergency sequence

1) Do not refresh repeatedly. 2) Open OPSOS from a clean tab. 3) Identify route with highest memory/freeze risk. 4) Enable scoped safe mode. 5) Verify. 6) Log. 7) Escalate if the issue remains.
