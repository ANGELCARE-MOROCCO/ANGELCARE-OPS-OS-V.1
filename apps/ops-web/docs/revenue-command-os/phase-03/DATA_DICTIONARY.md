# Data dictionary

The migration creates the following isolated Revenue OS domains:

- `revenue_os_knowledge_domains`: taxonomy and ownership.
- `revenue_os_doctrines`: canonical doctrine master records.
- `revenue_os_doctrine_sections`: structured content blocks.
- `revenue_os_doctrine_rules`: machine-readable rules.
- `revenue_os_doctrine_decision_trees`: governed branching logic.
- `revenue_os_knowledge_assets`: document and evidence registry.
- `revenue_os_knowledge_asset_versions`: asset version chain.
- `revenue_os_knowledge_chunks`: future retrieval chunks without external indexing.
- `revenue_os_knowledge_relationships`: evidence, implementation, contradiction and applicability graph.
- `revenue_os_sales_scripts`: approved channel scripts.
- `revenue_os_objection_patterns`: diagnostic objection handling.
- `revenue_os_case_studies`: success, failure, recovery and experiment memory.
- `revenue_os_campaign_patterns`: reusable campaign patterns.
- `revenue_os_playbooks`: executable sales and revenue playbooks.
- `revenue_os_policy_restrictions`: price, discount, legal, brand, privacy, promise and approval restrictions.
- `revenue_os_brand_requirements`: brand controls.
- `revenue_os_partner_benefits`: governed benefits and eligibility.
- `revenue_os_knowledge_approvals`: human approval dossiers.
- `revenue_os_knowledge_conflicts`: contradiction and resolution ledger.
- `revenue_os_knowledge_versions`: generic immutable resource snapshots.
- `revenue_os_knowledge_index_jobs`: controlled indexing queue.
- `revenue_os_knowledge_validation_issues`: validation findings.
- `revenue_os_knowledge_snapshots`: validation and readiness history.

All business records include ownership, status, source/provenance, versioning or update timestamps as applicable. Direct anonymous and authenticated client access is revoked; service-role access is used through the server repository.
